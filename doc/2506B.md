## Critical Issues

### 1. **Missing Date Validation in Controller**
**File:** `app/controllers/admin/reports_controller.rb:260-261`

The controller directly calls `.to_date` on user input without error handling. This will raise `Date::Error` exceptions for invalid input like "invalid" or "13/45/2025", causing application crashes.

**Fix Required:**
```ruby
begin
  @starting_date = params[:starting_date].to_date
  @ending_date = params[:ending_date].to_date
rescue Date::Error
  flash[:error] = "Invalid date format. Please use MM/DD/YYYY format."
  redirect_to admin_reports_path and return
end
```

### 2. **Nil Reference Error on Expiration Date**
**File:** `app/models/reports/flex_pass_patron_report.rb:38`

The code calls `strftime` on `flex_pass.expiration_date` without checking for nil, which will cause runtime errors if any FlexPass has a nil expiration date.

**Fix Required:**
```ruby
expiration_date: flex_pass.expiration_date&.strftime('%m/%d/%Y') || 'N/A',
```

## Recommendations

### `app/controllers/admin/reports_controller.rb`

#### Missing Date Range Validation
The controller lacks validation to ensure start date is before end date. Other reports in the same controller implement date swapping when the range is reversed. This should be added for consistency:

```ruby
if @starting_date > @ending_date
  @starting_date, @ending_date = @ending_date, @starting_date
end
```

#### Missing Authorization Check Before Database Query
While CanCan authorization is properly configured, the controller doesn't explicitly check authorization before executing the database query in the non-download path. Consider adding an explicit authorization check.

### `app/models/reports/flex_pass_patron_report.rb`

#### Hardcoded Table Name in Query
Lines 18-19 use hardcoded 'orders' table name in the WHERE clause. This could break if FlexPassOrder uses a different table name through inheritance:

```ruby
# Current (potentially fragile):
.where('orders.created_at >= ? AND orders.created_at <= ?', ...)

# Better:
.where("#{FlexPassOrder.table_name}.created_at >= ? AND #{FlexPassOrder.table_name}.created_at <= ?", ...)
```

#### Potential N+1 Query Issue
The `includes` statement may not prevent N+1 queries when accessing `flex_pass.uses_remaining` (line 39), as this method likely performs its own database query. Consider optimizing or preloading related data.

#### Missing Phone Number Formatting
Phone numbers are included raw without any formatting or validation. Consider standardizing phone number format for consistency.

### `app/models/ability.rb`

#### Authorization Group Inconsistency
The new action is added to `:reconciliation_reports`, but similar patron-focused reports are typically under `:box_office_reports`. This may cause confusion about access control patterns.

### `app/views/admin/reports/flex_pass_patron_report.html.haml`

### `config/routes.rb`
