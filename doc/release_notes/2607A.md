# Release Notes — July 2026 (Library Passes & Membership Management)

This release introduces **library passes** — staff-issued shared memberships — and a
major refresh of how memberships are managed and reported. It merges PR #15.

## New: Library Passes (timed memberships)

Membership offers now come in two types:

- **Production** memberships are the classic personal subscription — everything works
  as before.
- **Timed** memberships are *library passes*: shared passes your organization issues to
  a partner (a library, a school) whose patrons check them out. They are never sold on
  the public site and don't bill through Stripe.

What that means at the box office:

- **Issuing a pass**: timed offers show an **Issue Pass** button (on the offer's row and
  detail page) that creates the membership directly — no purchase order, no payment.
- **Redemption rules**: a library pass is good for one performance per calendar week
  (Monday through Sunday), up to its tickets-per-performance limit, and can only book
  performances in the current week. Clear messages explain when a pass has already been
  used or when booking opens.
- Every offer now shows **either** Create Order (production) **or** Issue Pass (timed) —
  greyed out while the offer is Inactive.

## New: Memberships list

**Passes > Memberships** (box office and admin users) lists every membership with live
search, sorting, and paging — no more scrolling one long page.

- **Columns**: Member Code (links to the membership), Offer, Member, Status, Start, and
  the new **Membership End**.
- **Start** shows when the membership actually began (the billing start date when there
  is one).
- **Membership End** shows when it ended — or, for a membership scheduled to cancel at
  period end, the final billing date with a *Cancel pending* label.
- **Default order** puts Active memberships first, then Suspended, then Canceled, with
  members alphabetical within each group. Click any column header to sort differently.
- **Search** matches member codes, member names, offer names, and statuses as you type.

## Membership Usage report improvements

- **Counts are accurate now.** Roughly 330 memberships canceled years ago had no
  recorded end date and were being counted as "active" in every month since — the report
  could show four times more active memberships than you really had. Their end dates
  have been reconstructed from their payment history, so monthly counts now reflect
  reality.
- **New "Members" column** counts *people* rather than memberships: a dual membership
  admits two members, so 27 dual memberships show as 54 members.
- **The bottom Total row no longer adds the count columns across months** (that
  double-counted every membership that lasted more than one month). It still totals the
  Collected and Paid money columns.

## Email list (MyEmma) sync

- When a membership becomes **Active**, the member's address is automatically added to
  the offer's MyEmma group; when it is **Canceled**, the address is removed — unless the
  patron still holds another current membership.
- **Changing the MyEmma Group ID on an offer** now re-syncs automatically: every active
  member's address is added to the new group. No manual list maintenance needed.
- Sync now runs in the background, so saving is instant and a temporary email-service
  problem can't block membership changes. Any sync problems are retried rather than
  silently dropped.
- The offer detail page displays the MyEmma group; the field appears only when MyEmma is
  configured for your installation.

## Smaller improvements

- The Passes menu has a new **Memberships** entry, and menu section dividers render
  correctly again.
- Membership offer detail pages show the offer name prominently and offer one-click
  **Edit** plus the offer's sales action.
- The user manual has a new [Managing Memberships](https://jwechsler.github.io/stagemgr/ticketing/managing-memberships/)
  page, and the Membership Offers and Membership Reports pages are updated with fresh
  screenshots.

## For administrators

- Deploying this release runs a one-time data migration that back-fills end dates on
  historically closed memberships.
- Restart the Resque workers after deploying so the new background sync jobs are
  recognized.
